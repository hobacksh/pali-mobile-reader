# 번역 워크플로우 (로컬 corpus 기준)

이 프로젝트는 `pali-mobile-reader/data/corpus`를 기준으로 번역합니다.

## 규칙

- 원문 경로: `data/corpus/romn/<file>.xml`
- 번역 경로: `data/corpus/ko/<file>.xml`
- **파일명은 원문과 동일**하게 유지합니다.
- 번역 파일은 원문 XML 구조를 기준으로 이미 정렬/복원되어 있어야 하며, 구조를 변경하지 않습니다.

## `trans` 키 기반 번역 규칙 (필수)

- 번역 작업은 `ko/<file>.xml`의 `trans` 속성을 기준으로 진행합니다.
- `trans="false"`: 미번역 상태. 이 항목만 번역 대상입니다.
- `trans="true"`: 번역 완료 상태. 재번역 대상이 아닙니다. (오탈자 수정 등 예외 작업은 별도)
- 번역 시 **태그 구조는 절대 변경하지 않고** 텍스트만 한국어로 교체합니다.
- 번역 완료 후 해당 태그의 `trans`를 `true`로 변경합니다.
- 미번역 항목(`trans=false`)은 원문(romn) 내용이 그대로 들어 있을 수 있으며, 이는 정상 상태입니다.
- 인라인 태그(`<pb>`, `<hi>`, `<note>` 등)의 종류/순서/속성은 원문과 1:1로 유지해야 합니다.
- `<note>`도 본문과 동일하게 번역 대상입니다. 주석/이본/출전 표기라고 해서 누락하면 안 됩니다.
- `<note>` 번역이 끝나면 해당 `<note>`의 `trans`도 반드시 `true`로 변경합니다.
- **라인(Line)의 정의**: 모든 작업 과정에서 언급되는 '라인'은 파일의 물리적 행 번호(1부터 시작하는 실제 줄 번호)를 의미합니다. XML 태그 내의 단락 번호(`n="..."`)와 혼동해서는 안 됩니다.

예:
- 원문: `data/corpus/romn/vin01m.mul.xml`
- 번역: `data/corpus/ko/vin01m.mul.xml`

## 에이전트 기반 번역 단계 (Manual/Agentic Process)

대형 언어 모델(LLM)을 사용하여 매뉴얼하게 번역을 진행할 경우 다음 단계를 따릅니다.

1. **로그 확인**: `data/corpus/ko/log/translation_log.md`를 읽어 현재 진행률을 확인합니다.
2. **청크 읽기**: `data/corpus/ko/chunks/` 디렉토리에서 **한 번에 딱 2개의 청크만** `view_file`로 읽습니다. (정밀도 유지 및 요약 방지를 위해 2개 단위를 고수함)
3. **내용 분석 및 원문 대조**:
   - **요약 금지**: 어떠한 경우에도 원문의 내용을 요약하거나 생략(괄호 처리 등)해서는 안 됩니다.
   - **수작업 직역만 허용**: 팔리어 원문의 문장 구조와 정보량을 그대로 한국어로 옮기는 수작업 직역만 수행합니다.
   - **대상 필터링**: `trans="false"` 항목만 번역 대상으로 선택합니다.
   - **동일 절/유사 절 대조 금지**: 번역 중 다른 절이나 기존 번역문과의 패턴 매칭으로 자동 치환하지 않습니다. 현재 항목을 독립적으로 직역합니다.
   - **구조 확인**: 원문의 단락(`<p>`), 강조(`<hi>`), 페이지 정보(`<pb>`) 등 모든 XML 태그의 위치와 속성을 확인합니다.
4. **전량 출력**: 출력 토큰 제한으로 인한 누락을 방지하기 위해, 번역할 양이 많다면 반드시 여러 번의 응답으로 나누어 **원문의 모든 문장을 빠짐없이 번역**하여 출력합니다.
5. **즉시 반영 (Immediate Application)**: 번역된 내용이 정밀성 규칙(요약 금지, 태그 보존)을 통과했다면, 사용자에게 다시 묻지 않고 즉시 대상 XML 파일에 반영하고 로그를 업데이트합니다.
   - 반영 시 `trans="false"` → `trans="true"`로 상태를 변경합니다.
6. **로그 업데이트**: 번역 일시, 청크 범위, 주요 내용을 로그 파일에 기록합니다.
7. **배치 운영 규칙**: 미번역 항목이 많을 때(예: 83개) 한 회차는 `2개 항목 × 5배치 = 총 10개 항목`까지만 처리합니다.

## 누락 방지 및 품질 관리 (Omission Prevention)

- **태그 및 구조 완전 보존 (Full XML Integrity)**: 원문의 모든 XML 태그(`<p>`, `<pb>`, `<hi>`, `<note>` 등)를 **1:1로 보존**합니다. 태그의 위치나 속성이 번역 과정에서 변형되어서는 안 됩니다.
- **주석 누락 금지**: `<note>`를 포함한 모든 텍스트 노드는 누락 없이 번역합니다. 출전 약호(예: `dha. pa.`, `saṃ. ni.`), 이본 표기(예: `(sī.)`, `(syā.)`)도 한국어 설명과 함께 유지합니다.
- **요약 및 괄호 처리 금지**: 원문이 반복적이거나 방대하다는 이유로 "~~에 대한 길고 상세한 설명" 식으로 요약하거나 괄호 안에 생략하는 행위는 **절대 금지**됩니다.
- **절 번호(§) 및 문단 대조**: 번역된 각 문단이 원문의 절 번호 및 문단 속성과 정확히 일치하는지 전수 조사합니다.
- **중략(`...pe...`) 표시**: 원문 XML에 있는 `...pe...` 기호는 원문이 생략된 형식을 그대로 나타내는 것이므로, 번역문에서도 이 형식을 유지하되 인위적인 추가 생략은 하지 않습니다.

## 파일 간 일관성 및 확장 (Multi-file Consistency)

현재는 `vin01m.mul.xml`을 번역하고 있으나, 다른 팔리 성전 파일(예: `vin02m...`)로 확장될 때도 동일한 품질을 유지하기 위해 다음을 준수합니다.

- **범용 용어 사전(Glossary) 준수**: 파일이 바뀌어도 핵심 불교 전문 용어와 관용구(예: '상인법', '거츠루장', '악작' 등)의 번역어는 프로젝트 전체에서 일관되어야 합니다.
- **로그의 정보 명확화**: 
  - `translation_log.md` 기록 시, 어떤 파일의 몇 번째 청크인지 파일명을 명시하여 소속을 명확히 합니다.
- **청크 관리**: 
  - 각 원문 파일에 해당하는 청크 디렉토리를 확인하고, 파일 switch 시 이전 파일의 상태와 섞이지 않도록 주의합니다.
- **파일별 특이사항 기록**: 특정 파일에만 나타나는 고유한 태그 형식이나 패턴이 있을 경우, 이를 워크플로우나 로그에 기록하여 일관성을 유지합니다.

## 메인 번역 프로세스: 에이전트 협업 방식 (Interactive Agent Process)

이 프로젝트는 자동화 스크립트보다는 **AI 에이전트와의 실시간 협업**을 통한 정밀 번역을 우선합니다. 다른 세션이나 다른 작업자가 참여하더라도 다음의 인터랙티브 프로세스를 유지해야 합니다.

1. **컨텍스트 로딩**: 새로운 세션 시작 시 에이전트에게 `TRANSLATION_WORKFLOW.md`와 `translation_log.md`를 먼저 읽게 하여 프로젝트 규칙과 현재 상태를 동기화합니다.
2. **청크 단위 작업**: 한 번에 **딱 2개의 청크**만 읽고 번역합니다. 이는 에이전트가 긴 텍스트를 처리하다가 지쳐서 내용을 요약해버리는 현상을 원천적으로 차단하기 위함입니다.
3. **태그 보존 중심**: 텍스트보다 XML 구조(태그)의 보존을 최우선으로 하며, 번역문 삽입 시 `cat`이나 `replace_file_content` 도구를 사용하여 원본 파일의 무결성을 지킵니다.
4. **자동 반영 루틴**: 번역 후 사용자의 승인을 기다리지 않고 직접 `run_command`나 `replace_file_content`를 통해 파일 수정을 완료합니다. 단, 결과물에 대한 사후 보고는 명확히 수행합니다.
